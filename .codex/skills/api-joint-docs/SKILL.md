# Agent 接口联调文档生成规范

## 1. 目的

当用户要求整理、生成或更新前后端联调接口文档时，Agent 输出一份简洁的 Markdown 接口契约。只保留联调必需内容：接口是什么、怎么请求、参数怎么校验、返回什么、错误怎么处理、这是新接口还是变更接口。

Agent 不默认以前端或后端为主。前端实现、后端实现、需求材料和旧文档都可以作为来源。

## 2. 参考文件

- 生成文档时使用 `references/api-doc-template.md`。
- 需要设计判断时读取 `references/guideline-digest.md`。
- 不强制生成 OpenAPI。

## 3. 来源

可使用这些来源：

- 前端：调用封装、页面字段、表单校验、Mock、TypeScript 类型、请求示例。
- 后端：路由、Controller、DTO、Schema、Validator、响应类型、错误码。
- 需求：PRD、任务描述、字段说明、交互说明。
- 旧文档：Markdown、Apifox、OpenAPI、Postman、历史接口说明。

处理规则：

- 只有需求时，生成草案，缺失内容写 `待确认`。
- 已有实现时，按实现提取契约，并简短记录来源。
- 多来源冲突时，不自动定稿，列入待确认。
- 不确定的字段不要编造。

## 4. 工作流

1. 用 `rg` 搜索路由、调用点、类型、校验器、Mock、接口名、path、需求关键词和历史文档。
2. 标记变更类型：`new`、`update`、`deprecated` 或 `待确认`。
3. 提取 method、path、auth、content-type、参数、响应、错误码。
4. 参数按 `path`、`query`、`header`、`cookie`、`body` 标明位置。
5. 每个参数至少写字段、位置、类型、必填、约束、示例、说明。
6. 更新接口必须说明兼容性影响和前端迁移点。
7. 使用模板生成 Markdown，并把冲突或缺失内容放到“待确认”。

## 5. 输出要求

- 文档要短，避免堆空表和无用占位。
- 路由优先资源化表达；项目已有风格优先。
- body 字段用点路径表达嵌套字段，例如 `user.name`。
- 错误响应至少包含状态码、业务码、触发条件、前端处理。
- 完成反馈包含文档路径、变更类型、来源、待确认项。
